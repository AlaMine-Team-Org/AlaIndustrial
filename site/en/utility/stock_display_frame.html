Stock Display Frame

alaindustrial:stock_display_frame
UtilityLVImplementedstoragedisplayitem-frameutilitylogistics

Summary

The Stock Display Frame is an entity frame (a subclass of the vanilla item frame) that hangs on any container — a vanilla chest, barrel, or shulker box, or the mod's Iron Chest — and shows the item count inside it right on the frame, in real time. It uses no energy: this is a quality-of-life logistics feature — "price tags" for rows of chests, so you can read a warehouse without opening a single GUI. Counting rule (the genre convention — AE2 Storage Monitor, Refined Storage; finalized by the 2026-07-12 playtest): an item is in the frame → the number = how many items of that type are in the container (a filter); the frame is empty → no number is shown (an empty frame looks like a plain frame; the server still computes the "everything" total — the comparator uses it); no container behind the frame → no number; the container is empty (with a filter in the frame) → shows 0. Only top-level slots are counted — no recursion into shulkers/bundles inside the chest. A double chest counts as one combined container (via ChestBlock.getContainer); a frame on either half shows the combined total.

Mechanics

Placing and removing

The frame item is placed with a right click on a block face (vanilla hanging-entity placement). The frame survives by the vanilla HangingEntity.survives rules: break the supporting block and the frame pops off, dropping its own item (not a vanilla item frame). Hitting a filled frame knocks the filter item out (first hit); a second hit removes the frame — inherited vanilla behaviour.

Interaction (right click)

Different from the vanilla frame — there is no item rotation (it is noise for a filter):

Emptyput the filter item in (1 pc) → the number appearsnothing

Maps (minecraft:filled_map) cannot be placed in the frame (none of the vanilla map-frame logic is engaged; filtering maps in a chest is meaningless).

Counting (server side)

Every stockFrameScanIntervalTicks (default 20 ticks = 1 s, C) the frame scans the block behind it: getPos.relative(getDirection.getOpposite). A level.isLoaded(targetPos) guard runs first: the chest can sit in an unloaded chunk while the frame's chunk is ticking; the scan never forces synchronous chunk loads. resolution: a ChestBlock goes through ChestBlock.getContainer(...) (merges double chests); anything else matches by BlockEntity instanceof (covers the barrel, shulker box, and third-party containers). Sums ItemStack.getCount across slots; with a filter, only stacks matching the filter item by type and components (ItemStack.isSameItemSameComponents). The result is a synced field (SynchedEntityData, int; -1 = "no container"). It is written only when the value changes — dirty entity data reaches clients immediately regardless of the entity's updateInterval`.

Displaying the number (client side)

The number is drawn flat on the frame's face, in its bottom strip — the sign-text idiom (SubmitNodeCollector.submitText + a scale(s, −s, s) pose, like AbstractSignRenderer). The text turns with the frame (not a billboard); white with a dark outline, no backdrop. Coordinate gotcha: the frame's local space is rotated 180° around Y relative to a sign's text space (+z points INTO the wall) — the text needs a YP(180°) turn plus a +0.03 push toward the viewer, otherwise the digits end up behind the backboard facing the wall. Constants (StockDisplayFrameRenderer): TEXT_SCALE = 0.02 (8px glyphs → 0.16 blocks), TEXT_Y = −0.17 (digit center below the frame center), ITEM_RAISE = 0.08 + ITEM_SCALE_WITH_COUNT = 0.4 — the filter item is raised and shrunk (0.5 → 0.4) to free the bottom strip; without a filter the frame renders like vanilla (item centered, scale 0.5). Display distance — up to 24 blocks (the text hides beyond that; the storage-mod FPS convention). Formatting: the exact number below 10,000, abbreviated after that — 12.3k, 1.2M. An empty frame or "no container" (-1) renders no text at all.

Redstone

getAnalogOutput is overridden: a comparator reads the container fullness behind the frame using the vanilla container curve (AbstractContainerMenu.getRedstoneSignalFromContainer — the same 0–15 curve as a comparator on the chest itself), not the vanilla frame-rotation signal (rotation is always 0 on this frame and would pin the output to a meaningless constant). The signal ignores the filter — it is the whole container's fullness. No container / unloaded chunk → 0.

Recipes

Crafting: data/alaindustrial/recipe/stock_display_frame.json Type: minecraft:crafting_shaped, category redstone. 3×3 pattern: GWG / SFS / GWG — G = alaindustrial:gold_dust, W = alaindustrial:wooden_gear, S = alaindustrial:silver_ingot, F = minecraft:item_frame (a vanilla item frame in the center). The idea: an ordinary frame is upgraded into a counting one with precise mechanics (gears) and precious electrics (gold dust + silver). Result: 1 × alaindustrial:stock_display_frame. A recipe-unlock advancement (recipe_unlocked) fires on obtaining a wooden_gear OR a silver_ingot (the mod convention: without the advancement the recipe is hidden from the recipe book).

Compatibility

Works with any block whose BlockEntity implements : vanilla chest / trapped chest / copper chest / barrel / shulker box, the mod's Iron Chest, third-party containers. Does not count inventories exposed only through the Transfer API / capabilities without a (a deliberate v1 simplification). Hoppers/pipes do not interact with the frame (it is an entity).

Implementation

Java (common): entity/StockDisplayFrameE — extends ItemFrame: constructors (EntityType, Level) + (EntityType, Level, BlockPos, Direction); the synced count; tick scanning every N ticks; overrides: interact (put/take the filter, no rotation, maps rejected), getFrameItemStack (own drop/pick-block), getAnalogOutput (container fullness). client/StockDisplayFrameR — extends ItemFrameRenderer: re-implements submit (frame + raised item + flat count text), sign-text idiom for the digits. item/StockDisplayFrameI — own useOn: the vanilla HangingEntityItem#useOn hardcodes which entity it spawns and silently spawns nothing for custom types. **Registration** (the: two handles in (the EntityType + the item), new registries ModEntities (Fabric, eager) / ModEntitiesNeoForge (DeferredRegister; the item resolves the type lazily inside the registration lambda) + one line each in /ModItemsNeoForge + the CreativeTabContent tab. Renderers: Fabric EntityRenderers.register(...); NeoForge EntityRenderersEvent.RegisterRenderers. **Assets:** textures/item/stock_display_frame.png — an original icon (a light frame with a counter badge). The in-world entity model is **custom geometry in the renderer** (renderFrameGeometry, submitCustomGeometry): the same element boxes and UVs as the vanilla models/block/template_item_frame.json, but the border bars sample the mod's own textures/block/stock_display_frame_wood.png texture (quartz with golden corners, original) while the backboard keeps the vanilla block/item_frame sprite (referenced at runtime — no Mojang assets are copied). The border texture is stitched into the block atlas via assets/minecraft/atlases/blocks.json (atlas JSONs merge additively across packs). The vanilla model cannot be re-textured — BlockModelResolver.updateForItemFrame hardwires the minecraft:item_frame fake blockstate — so state.frameModel is cleared in extract and the geometry is drawn by the renderer itself. Lang: item.alaindustrial.stock_display_frame + entity.alaindustrial.stock_display_frame` (en_us is the source; all 20 locales).

Testing

Gametests (the shared scenarios in common/.../gametest/StockDisplayFrameS, wrapped by the Fabric suite + registered in NeoForgeGameTests), suite TC-FRAME-001: a frame on a chest with 5 diamonds + 3 iron, no filter → server count 8 (comparator source); a diamond filter with diamonds + iron in the chest → counts only diamonds; a double chest → the sum of both halves; adding items to the chest → the count updates within one scan interval; a frame on a non-container → count −1; breaking the frame → drops the stock_display_frame` item (not a vanilla frame).

Open questions

A glow version. Not in v1; if players struggle to read the digits in dark rooms, a glow tier can come later. A second text line. Deferred to keep the frame clean.

Progression

LV tier: available after the first gears and silver (recipe: a vanilla item frame + wooden_gear ×2 + silver_ingot ×2 + gold_dust ×4).
